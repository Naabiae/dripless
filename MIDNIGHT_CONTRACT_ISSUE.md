# MIDNIGHT_CONTRACT_ISSUE.md — ProofRail Midnight Smart Contracts

## What Is This?

This document covers the complete build of the **Midnight Network smart contract layer** for ProofRail — the Privacy-first P2P fiat-to-crypto on/off-ramp. The backend (NestJS) is already done. This layer is the on-chain half that the backend integrates with.

There are **two Compact contracts** and a **TypeScript DApp driver** that wires them to the backend:

| Contract | File | Purpose |
|---|---|---|
| KYC Credential Verifier | `proof_rail_kyc.compact` | Verifies a credential issued by the backend. Returns a boolean on-chain proof. |
| P2P Escrow | `proof_rail_escrow.compact` | Holds funds in shielded escrow. Releases to buyer or seller on backend instruction. |

The TypeScript DApp driver (`src/midnight/`) lives **inside the backend repo**. It calls the compiled contract circuits from NestJS service methods. This is how backend and chain stay in sync — the driver is the bridge.

---

## Key Things to Understand Before Writing a Single Line

### Compact is not Solidity

Compact is a TypeScript-style DSL that compiles to ZK circuits. Critical differences:

- **No `msg.sender`** — exposing the sender would break privacy. Identity is proven via witnesses (private data) and `persistentHash`.
- **No on-chain execution** — circuits run locally on the user's machine. A ZK proof is generated then submitted. Validators verify the proof, not the computation.
- **`disclose()` is mandatory** — any witness-derived value written to the public ledger must be wrapped in `disclose()`. The compiler rejects programs that forget this.
- **Witnesses are off-chain callbacks** — `witness getCredentialHash(): Bytes<32>` is implemented in TypeScript and runs locally. The output is used in a circuit but never leaves the machine as raw data.
- **`ledger` = public on-chain state** — every `ledger` field is visible to all.
- **`Counter`** — Midnight's standard library type for round counters, used to prevent replay attacks and maintain anonymity (changing the round means the public key hash changes each circuit call).

### The Proof Server is Required

Every transaction that calls a circuit must generate a ZK proof. This requires a **proof server** running as a Docker container. In development, this runs locally. In production, it runs alongside the backend.

```
Docker container: midnight-proof-server (from official image)
Port: 6300
Must be running before any circuit call
```

```
        │
        ▼

        ▼
TypeScript DApp driver (midnight-kyc.service.ts):
  1. Loads compiled proof_rail_kyc contract
  2. Constructs witness from backend credential payload
  3. Calls verifyCredential() circuit locally
  4. Proof server generates ZK proof
  5. Transaction submitted to Midnight Preprod/Mainnet
  6. Returns tx hash to NestJS KYC module
        │
        ▼

        │
        ▼

        │
        ▼

  1. Loads compiled proof_rail_escrow contract
  2. Verifies seller holds valid KYC credential on-chain
  3. Calls lockFunds(tradeId, amount) circuit

  5. Returns escrow tx hash → backend stores it

        ▼

        │
        ▼
```

## Issue SC-2 — KYC Credential Verifier Contract (`proof_rail_kyc.compact`)

### Status: `[ ] Open`
### Depends On: Issue SC-1 ✅

### What This Is

The first real contract. It verifies that a user holds a valid ProofRail credential (issued by the NestJS backend in Issue 4). The contract does **not** store raw KYC data. It stores a hash of the credential's key claims, plus a registry of verified wallet addresses. When a user calls `verifyCredential()`, the circuit:

1. Takes the credential claims as a private witness
2. Hashes them with the backend's HMAC signature as a commitment
3. Asserts the hash matches the one registered by the backend
4. Marks the wallet address as `VERIFIED` on the public ledger

The public ledger only ever shows: `address → VERIFIED/UNVERIFIED`. Nothing else.

### Research Notes

- `persistentHash<T>(value: T): Bytes<32>` — hashes data for use as a key. Stable across rounds, used for identity commitments.
- `transientCommit<T>(value: T, rand: Field): Field` — commitment scheme with random nonce. Use this to commit to credential data without revealing it.
- The `disclose()` requirement: the verified status written to the ledger is derived from witness data (the credential). Must be wrapped in `disclose()`.
- `Map<K, V>` is available from `CompactStandardLibrary`. Used to store `walletAddress → status` mapping.
- The backend issues a credential with an HMAC signature (`CREDENTIAL_SECRET`). The contract verifies by checking that `hash(claims || signature)` matches what was registered. The backend calls `registerCredential()` to write the expected hash on-chain first, then the user calls `verifyCredential()` to prove they hold the matching claims.
- `Opaque<"string">` is used for wallet addresses since they're JavaScript strings passed in from the DApp driver.
- Counter/round is included for replay protection — a fresh `instance` value each circuit call prevents the same proof from being replayed.

### Contract Design

```
Public Ledger (what's on-chain, visible to all):
  - credentialRegistry: Map<Bytes<32>, CredentialStatus>
    Key: persistentHash(walletAddress)
    Value: { status: UNREGISTERED/REGISTERED/VERIFIED, expiresAt: Uint<64> }
  - round: Counter  (replay protection)

Private (witness data, never leaves user's machine):
  - getCredentialClaims() → claims struct
  - getCredentialSignature() → HMAC signature bytes
  - getWalletKey() → user's private key (for address derivation)

Circuits:
  - registerCredential(commitmentHash, expiresAt)
    → called by backend service after issuing credential
    → writes commitment to credentialRegistry

  - verifyCredential()
    → called by user's DApp
    → reconstructs commitment from witness claims + sig
    → asserts it matches registered commitment
    → marks entry as VERIFIED

  - isVerified(walletAddressHash): Boolean
    → pure circuit (no ledger write)
    → read-only check used by escrow contract

  - revokeCredential(walletAddressHash)
    → called by backend service
    → sets status to UNREGISTERED
```

### Contract Code (`contracts/proof_rail_kyc.compact`)

```compact
pragma language_version 0.28;
import CompactStandardLibrary;

enum CredentialStatus { UNREGISTERED, REGISTERED, VERIFIED }

struct CredentialEntry {
  status: CredentialStatus;
  expiresAt: Uint<64>;
  commitment: Field;
}

export ledger credentialRegistry: Map<Bytes<32>, CredentialEntry>;
export ledger round: Counter;

// Witnesses — private data provided by DApp driver, never on-chain
witness getClaimsCommitment(): Field;
witness getWalletAddressHash(): Bytes<32>;

// Called by backend service to register a new credential commitment
export circuit registerCredential(
  walletAddressHash: Bytes<32>,
  commitment: Field,
  expiresAt: Uint<64>
): [] {
  const entry = CredentialEntry {
    status: CredentialStatus.REGISTERED,
    expiresAt: disclose(expiresAt),
    commitment: disclose(commitment),
  };
  credentialRegistry.insert(walletAddressHash, entry);
}

// Called by user to prove they hold matching claims
export circuit verifyCredential(): [] {
  const addrHash = disclose(getWalletAddressHash());
  const claimsCommitment = getClaimsCommitment();

  const entry = credentialRegistry.lookup(addrHash);
  assert(entry is_some, "Credential not registered");

  const e = entry.value;
  assert(e.status == CredentialStatus.REGISTERED, "Already verified or revoked");
  assert(e.commitment == claimsCommitment, "Claims do not match registered commitment");
  assert(e.expiresAt > round as Uint<64>, "Credential expired");

  const updatedEntry = CredentialEntry {
    status: CredentialStatus.VERIFIED,
    expiresAt: e.expiresAt,
    commitment: e.commitment,
  };
  credentialRegistry.insert(addrHash, updatedEntry);
  round.increment(1 as Uint<64>);
}

// Pure read — used by escrow contract to gate trade entry
export circuit isVerified(walletAddressHash: Bytes<32>): Boolean {
  const entry = credentialRegistry.lookup(walletAddressHash);
  if (entry is_some) {
    return entry.value.status == CredentialStatus.VERIFIED
      && entry.value.expiresAt > round as Uint<64>;
  }
  return false;
}

// Called by backend admin to revoke
export circuit revokeCredential(walletAddressHash: Bytes<32>): [] {
  const entry = credentialRegistry.lookup(walletAddressHash);
  assert(entry is_some, "Credential not found");
  const e = entry.value;
  const revokedEntry = CredentialEntry {
    status: CredentialStatus.UNREGISTERED,
    expiresAt: e.expiresAt,
    commitment: e.commitment,
  };
  credentialRegistry.insert(walletAddressHash, revokedEntry);
}
```

> **⚠️ Note:** Compact syntax evolves. Always run `compact compile contracts/proof_rail_kyc.compact contracts/managed` and fix any errors the compiler surfaces. The Midnight MCP server in your AI assistant will catch these before you run the compiler.

## Issue SC-3 — P2P Escrow Contract (`proof_rail_escrow.compact`)

### Status: `[ ] Open`
### Depends On: Issue SC-2 ✅

### What This Is

The escrow contract holds the seller's crypto in a shielded state until the trade completes. The contract enforces:

- Only KYC-verified users can initiate escrow (cross-contract call to KYC contract's `isVerified`)
- Funds are locked in a shielded escrow record keyed by `tradeId`
- Only the registered seller can lock funds
- Funds release to buyer on backend instruction (signed release instruction)
- Funds return to seller if disputed and admin resolves in seller's favor
- Funds are never visible on-chain — amounts are shielded

### Research Notes

- Midnight uses a **UTXO model** for native NIGHT/DUST tokens. For trading app escrow, you would work with **contract tokens** (account-based, like ERC-20 adapted for Compact).
- Cross-contract calls: to call `isVerified()` from the KYC contract within the escrow contract, you import the KYC contract's verification key (its address on-chain acts as its identity). This is done via `import contract proof_rail_kyc;` (exact syntax depends on Compact version — verify with MCP server).
- For MVP: escrow uses `Opaque<"string">` amounts (backend-verified) + a commitment. Real token lock using `@midnight-ntwrk/compact-runtime` token APIs comes in production.
- `tradeId` as an `Opaque<"string">` is the escrow key. The ledger maps `persistentHash(tradeId)` → `EscrowEntry`.
- Release instructions from the backend are signed with `MIDNIGHT_RELEASE_SECRET`. The contract verifies the signature commitment before releasing.

### Contract Design

```
Public Ledger:
  - escrowRegistry: Map<Bytes<32>, EscrowEntry>
    Key: persistentHash(tradeId)
    Value: { status, sellerKeyHash, buyerKeyHash, amountCommitment, expiresAt }
  - round: Counter

Private (witness):
  - getTradeId(): Opaque<"string">
  - getSellerKey(): Bytes<32>
  - getBuyerKey(): Bytes<32>
  - getReleaseSignature(): Bytes<32>

Circuits:
  - lockFunds(tradeId, buyerKeyHash, amountCommitment, expiresAt)
    → seller calls this
    → verifies seller is KYC-verified (isVerified check)
    → creates escrow entry LOCKED

  - releaseToBuyer(tradeId, releaseSignature)
    → backend service calls this after seller confirms payment
    → verifies signature (backend-signed instruction)
    → sets entry to RELEASED_BUYER

  - releaseToSeller(tradeId, releaseSignature)
    → admin calls this for dispute resolution in seller's favor
    → verifies signature
    → sets entry to RELEASED_SELLER

  - cancelEscrow(tradeId)
    → seller calls this, only valid if still LOCKED and not expired
    → sets entry to CANCELLED

  - getEscrowStatus(tradeIdHash): EscrowStatus
    → pure read circuit
```

### Contract Code (`contracts/proof_rail_escrow.compact`)

```compact
pragma language_version 0.28;
import CompactStandardLibrary;

enum EscrowStatus { LOCKED, RELEASED_BUYER, RELEASED_SELLER, CANCELLED, EXPIRED }

struct EscrowEntry {
  status: EscrowStatus;
  sellerKeyHash: Bytes<32>;
  buyerKeyHash: Bytes<32>;
  amountCommitment: Field;     // commitment to trade amount, never revealed
  expiresAt: Uint<64>;
}

export ledger escrowRegistry: Map<Bytes<32>, EscrowEntry>;
export ledger round: Counter;

// Witnesses
witness getSellerKeyHash(): Bytes<32>;
witness getBuyerKeyHash(): Bytes<32>;
witness getTradeIdHash(): Bytes<32>;
witness getReleaseSignatureHash(): Field;

// Seller locks funds — creates escrow entry
export circuit lockFunds(
  tradeIdHash: Bytes<32>,
  buyerKeyHash: Bytes<32>,
  amountCommitment: Field,
  expiresAt: Uint<64>
): [] {
  const sellerKeyHash = disclose(getSellerKeyHash());
  
  const existingEntry = escrowRegistry.lookup(tradeIdHash);
  assert(existingEntry is_none, "Trade already has escrow");

  const entry = EscrowEntry {
    status: EscrowStatus.LOCKED,
    sellerKeyHash: sellerKeyHash,
    buyerKeyHash: disclose(buyerKeyHash),
    amountCommitment: disclose(amountCommitment),
    expiresAt: disclose(expiresAt),
  };

  escrowRegistry.insert(tradeIdHash, entry);
  round.increment(1 as Uint<64>);
}

// Backend-signed instruction releases to buyer
export circuit releaseToBuyer(
  tradeIdHash: Bytes<32>,
  releaseSignatureHash: Field
): [] {
  const entry = escrowRegistry.lookup(tradeIdHash);
  assert(entry is_some, "Escrow not found");

  const e = entry.value;
  assert(e.status == EscrowStatus.LOCKED, "Escrow not in LOCKED state");
  assert(e.expiresAt > round as Uint<64>, "Escrow expired");
  assert(transientCommit(releaseSignatureHash, round as Field) 
    == getReleaseSignatureHash(), "Invalid release signature");

  const updatedEntry = EscrowEntry {
    status: EscrowStatus.RELEASED_BUYER,
    sellerKeyHash: e.sellerKeyHash,
    buyerKeyHash: e.buyerKeyHash,
    amountCommitment: e.amountCommitment,
    expiresAt: e.expiresAt,
  };
  escrowRegistry.insert(tradeIdHash, updatedEntry);
  round.increment(1 as Uint<64>);
}

// Admin-signed instruction releases to seller (dispute won by seller)
export circuit releaseToSeller(
  tradeIdHash: Bytes<32>,
  releaseSignatureHash: Field
): [] {
  const entry = escrowRegistry.lookup(tradeIdHash);
  assert(entry is_some, "Escrow not found");

  const e = entry.value;
  assert(e.status == EscrowStatus.LOCKED, "Escrow not in LOCKED state");
  assert(transientCommit(releaseSignatureHash, round as Field)
    == getReleaseSignatureHash(), "Invalid release signature");

  const updatedEntry = EscrowEntry {
    status: EscrowStatus.RELEASED_SELLER,
    sellerKeyHash: e.sellerKeyHash,
    buyerKeyHash: e.buyerKeyHash,
    amountCommitment: e.amountCommitment,
    expiresAt: e.expiresAt,
  };
  escrowRegistry.insert(tradeIdHash, updatedEntry);
  round.increment(1 as Uint<64>);
}

// Seller cancels (only in LOCKED state, before expiry)
export circuit cancelEscrow(tradeIdHash: Bytes<32>): [] {
  const sellerKeyHash = getSellerKeyHash();
  const entry = escrowRegistry.lookup(tradeIdHash);
  assert(entry is_some, "Escrow not found");

  const e = entry.value;
  assert(e.status == EscrowStatus.LOCKED, "Cannot cancel non-LOCKED escrow");
  assert(e.sellerKeyHash == sellerKeyHash, "Only seller can cancel");

  const updatedEntry = EscrowEntry {
    status: EscrowStatus.CANCELLED,
    sellerKeyHash: e.sellerKeyHash,
    buyerKeyHash: e.buyerKeyHash,
    amountCommitment: e.amountCommitment,
    expiresAt: e.expiresAt,
  };
  escrowRegistry.insert(tradeIdHash, updatedEntry);
  round.increment(1 as Uint<64>);
}

// Pure read — used by backend driver to check status
export circuit getEscrowStatus(tradeIdHash: Bytes<32>): EscrowStatus {
  const entry = escrowRegistry.lookup(tradeIdHash);
  if (entry is_some) {
    return entry.value.status;
  }
  return EscrowStatus.CANCELLED; // default for not-found
}
```

> **⚠️ Note:** Run `compact compile` on this immediately after writing. The `transientCommit` signature assertion pattern especially — verify it compiles correctly with your Compact version. Use the MCP server to get the exact commit API for your compiler version.

### NestJS Driver (`src/midnight/midnight-escrow.service.ts`)

```typescript
// Key methods:

// lockFunds(tradeId, buyerWalletAddress, amountUsd)
//   → compute tradeIdHash, buyerKeyHash, amountCommitment
//   → call lockFunds circuit with seller as the witness provider
//   → return tx hash → backend stores in trade.escrowTxHash

// releaseToBuyer(tradeId)
//   → backend generates release signature (HMAC of tradeId + 'BUYER')
//   → construct releaseSignatureHash witness
//   → call releaseToBuyer circuit
//   → return tx hash

// releaseToSeller(tradeId)
//   → same as above, direction 'SELLER'
//   → used by disputes.service after admin resolution
```

## Critical Reminders

1. **Compact syntax changes per version.** Always check `compact --version` and verify syntax with the Midnight MCP server before assuming any code snippet is correct.

2. **The proof server must be running for any circuit call.** If `localhost:6300` is down, all Midnight transactions will fail silently or with a confusing network error. Add a health check in `provider.service.ts` that pings the proof server on startup.

3. **Preprod tokens needed.** The backend wallet needs tNIGHT tokens to generate tDUST for transaction fees. Fund it at: `https://faucet.preprod.midnight.network` before running any contract deployment.

4. **ZK proof generation is slow (~5-30 seconds per transaction).** Design your UI and backend timeouts accordingly. Never set a 5-second timeout on Midnight transactions.

5. **Contracts are currently immutable on Midnight.** Deploy carefully. If you find a bug after deployment, you deploy a new contract at a new address and migrate.

6. **Backend wallet ≠ user wallets.** The NestJS backend service has its own funded wallet for calling admin circuits (`registerCredential`, `revokeCredential`, `releaseToBuyer`, `releaseToSeller`). User wallets interact via the frontend DApp driver — this is separate from the backend wallet.
